## 投稿1（速報型）【約310文字】
> ソース: [Manage agent skills with GitHub CLI - GitHub Changelog](https://github.blog/changelog/2026-04-16-manage-agent-skills-with-github-cli/)
> 推奨投稿時刻: 09:00

`gh skill` が来た。

GitHub CLI v2.90.0 にAgent Skillsのパッケージマネージャーが入った。npmやHomebrewと同じ感覚で、GitHubのリポジトリからスキルをインストール・更新・共有できる。

```
gh skill install github/awesome-copilot documentation-writer \
  --agent claude-code --scope user
```

タグ + git の tree SHA で管理されてるから、`gh skill update` は中身が変わったスキルだけ検出してくれる。`--pin v1.2.0` でバージョン固定もできる。Claude Code だけじゃなく Codex・Cursor・Copilot・Gemini CLI にも対応してるのが面白くて、エージェントをまたいでスキルを共有できる仕様になってる。

一点だけ注意: GitHubが検証してるわけじゃないのでプロンプトインジェクションのリスクはある。公式も「インストール前に `gh skill preview` で確認を」と明記してるので素性不明のスキルには注意。

使い捨てプロンプトからバージョン管理されたSkillへ、という流れが本格的に始まった気がする。

## 投稿2（解説型・CTA付き）【約430文字】
> ソース: [Enabling Claude Code to work more autonomously - Anthropic](https://www.anthropic.com/news/enabling-claude-code-to-work-more-autonomously)
> 推奨投稿時刻: 15:00

Anthropicが「Claude Codeをもっと自律的に動かす」という公式ビジョンをまとめたブログを出した。個々の機能は知ってたつもりだったけど、一本の記事で並べてみると「あ、この方向に振り切るつもりなんだ」という感覚がある。

大きな柱は3つ。

まずVS Code Extension（beta）。ターミナル外に出てサイドバーパネルでインラインdiffをリアルタイムで見ながら作業できる。「CLIのツールがIDEに移ってくる」のを公式が宣言した形。

次にCheckpointing。各変更前にコードの状態を自動保存して、Esc×2か `/rewind` で即時ロールバックできる。会話だけ・コードだけ・両方、と復元対象を選べる。これ、「AIに怖くて任せられない」という人の心理的ハードルをかなり下げると思う。自分もそう感じた。

そしてSubagents + Hooks の組み合わせ。バックエンドAPIをSubagentに投げながらメインエージェントがフロントを並列で組む、みたいな開発ワークフローが想定されてる。

「ターミナルのツール」から「IDEネイティブの自律エージェント」へ、という進化を公式が一枚絵にして出してきた感じがある。こういうエージェント設計の考え方を学べる場所 → https://claude-code-academy.dev/

## 投稿3（Tips型）【約340文字】
> ソース: [Claude Code /powerup Interactive Lessons Guide - claudefa.st](https://claudefa.st/blog/guide/mechanics/claude-powerup)
> 推奨投稿時刻: 21:00

`/powerup` 、使ってる人どのくらいいるんだろ。

Claude Code v2.1.90 から入ってる機能で、ターミナルを離れずにアニメーション付きのインタラクティブレッスンが18本受けられる。ドキュメントサイトを別タブで開かなくてもいい、というのが設計の意図らしい。VS Code でいうインタラクティブチュートリアルに近いポジション。

自分は「なんとなく動かせてる」状態でしばらく使ってたんだけど、/powerup を一周したら知らずに損してた機能がいくつかあった。完了済みレッスンはチェックマークで管理できるので途中で止めても戻れる。

全プラン・追加費用なし。Free ユーザーでも使える。

「Claude Code は触ってるけど全機能把握できてる気がしない」という人はまず `/powerup` を開いてみるといい。18本全部やっても重くないし、最初から通して見るより気になったレッスンをつまみ食いするくらいの使い方で十分だと思う。

## 投稿4（議論型）【約370文字】
> ソース: [Model selection for Claude and Codex agents on github.com - GitHub Changelog](https://github.blog/changelog/2026-04-14-model-selection-for-claude-and-codex-agents-on-github-com/)
> 推奨投稿時刻: 03:00

GitHub.com の Copilot 画面でタスクを作るとき、Claude Sonnet 4.6 か Opus 4.6 かを選べるようになった。

これ、地味に大きい変化で。「GitHub Copilot を使う」と「Claude Code を使う」の境界線がもうほとんどなくなってきてる。CopilotのBusiness・Enterpriseサブスクに含まれてるから、すでに契約してる人はそのまま管理者がポリシーを有効にすれば Claude のモデルでエージェントタスクを走らせられる。

Claude Code ユーザーとしては「じゃあ自分は何をターミナルで使う意味があるのか」という問いがちょっとよぎった。全部 GitHub の UI に移行するわけじゃないと思うんだけど、コンテキストやフック・サブエージェントを細かく制御したい用途はCLIに残って、「このPR直してきて」くらいのタスクは Web 側に流れていくのかな、と今は考えてる。

どちらにせよ Sonnet vs Opus の使い分けが GitHub の画面でも意識される時代になった。みんなはどっちを選んでる？
