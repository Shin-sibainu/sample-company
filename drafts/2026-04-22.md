## 投稿2（解説型・CTAなし）【約380文字】
> ソース: [Claude Code power user tips | Claude Help Center](https://support.claude.com/en/articles/14554000-claude-code-power-user-tips) / [CLAUDE.md Productivity Stack - Botmonster](https://botmonster.com/posts/claude-md-productivity-stack-custom-commands-git-worktrees-agent-rules/) / [Claude Code Best Practices - rosmur.github.io](https://rosmur.github.io/claudecode-best-practices/)
> 推奨投稿時刻: 15:00

CLAUDE.mdに色々書きまくってた時期があって、でもよく考えたら「これ守られてる？」って確認したことなかった。

調べてみると、CLAUDE.mdの指示が実行される確率は約70%らしい。フックは100%。

この差、知ってた人どれくらいいるんだろう。自分は普通に全部CLAUDE.mdで管理できると思い込んでた。「テスト必ず回してから」「このコマンドは絶対使うな」みたいなガードレール系も全部そこに書いてた。

それがフックに移すべきだったという話で、設定自体は `.claude/settings.json` の `hooks` キーに書くだけ。たとえばコミット前にテストを強制実行するなら

```
"PostToolUse": [{"matcher": "Bash", "hooks": [{"type": "command", "command": "npm test"}]}]
```

みたいな形になる。

CLAUDE.mdはClaude への「お願い」で、フックは「強制」。この使い分けを最初から知っていれば、ルールが守られない謎に悩む時間が減ったなと思う。

## 投稿3（Tips型）【約350文字】
> ソース: [Claude Code Worktrees: Parallel Sessions Without Conflicts](https://claudefa.st/blog/guide/development/worktree-guide) / [Parallel Development with ClaudeCode and Git Worktrees - Medium](https://medium.com/@ooi_yee_fei/parallel-ai-development-with-git-worktrees-f2524afc3e33)
> 推奨投稿時刻: 21:00

`claude -w` を使うとソロ開発が体感かなり変わる。

やることはシンプルで、タスクごとに独立したgit worktreeを作って、それぞれ別のClaude Codeセッションで並列実行する。「feature/auth」「fix/payment-bug」「refactor/api」とかを同時に走らせて、自分は通知が来たときだけ確認するだけ。

最初に試したとき、3セッション並列で「あ、これ小さなチームになった感じだ」と思った。自分がやるべきことは意思決定だけで、実装の大半は同時進行してる。

命名は `{型}/{機能名}` で統一しておくと、worktreeが増えてもどれが何かわからなくなる事態が防げる。`git worktree list` で一覧確認も習慣にしておくといい。

Notificationフックと組み合わせると「全部手放して通知だけ受け取る」運用になって、10〜15並列の実績もあるらしい。さすがにそこまでやるかは別として、3〜5本並列でも体感は結構変わる。

Reddit 500人規模の調査が面白かった。r/ClaudeAI（週間アクティブ4,200人）とr/Codex（同1,200人）を横断分析した記事。

結論は「どっちが勝ち」じゃなくて「場面で使い分ける派が多数」。Claude Codeはエージェント長時間タスクとコンテキスト保持と全体把握、CodexはVS Code統合の滑らかさと短いスニペット補完、という棲み分けに落ち着いてる感じ。

自分はClaude Code一本だけど、「VSCodeから出たくない」という気持ちはわかるし、補完の気持ちよさではCodexの方が上という話も否定できない。

なんというか、どっちが正しいかより「どのフローに合うか」で選ぶのが現実的だよなとは思う。みんなはどっち派、もしくは両刀？
