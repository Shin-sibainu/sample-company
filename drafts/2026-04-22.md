> ソース: [Claude Code Used to Find Remotely Exploitable Linux Kernel Vulnerability Hidden for 23 Years — InfoQ](https://www.infoq.com/news/2026/04/claude-code-linux-vulnerability/)
> 推奨投稿時刻: 09:00

23年間、誰も気づかなかった。

Anthropicの研究者Nicholas CarliniがClaude Code（Opus 4.6）を使って、LinuxカーネルのNFSドライバに2003年から眠っていたリモート悪用可能なヒープバッファオーバーフローを発見した。

しかも手法がシンプルすぎて逆に驚く。カーネル全ソースファイルをBashスクリプトで順番にClaude Codeに渡しただけ。それで500件超の高深刻度ゼロデイを掘り当てた。

Opus 4.1やSonnet 4.5では同じことをやってもほとんど再現できなかったらしく、4.6での能力ジャンプがいかに急だったかがわかる。

「AIはコードを書く道具」という認識のまま使ってるとこういうニュースで少し止まる。

## 投稿2（解説型）【約380文字】
> ソース: [Making frontier cybersecurity capabilities available — Anthropic](https://www.anthropic.com/news/claude-code-security)
> 推奨投稿時刻: 15:00

Claude Code Securityというプロダクトが存在するのを最近まで知らなかった。

普段使いのClaude Codeとは別の製品で、セキュリティ研究者向けのlimited research previewとして公開されている。何が違うかというと、コードを「パターンで照合する」のではなく「人間のセキュリティ研究者として読む」という動き方をする。SQLインジェクションや認証バイパスを複数ファイルにまたがって推論で見つける、という感じ。

Anthropicのチームが実際にOSSコードベースで検証して、500件超の高深刻度脆弱性を確認済みという話と、今日の速報のLinuxカーネル発見はつながってる。同じOpus 4.6ベースの推論能力がどちらにも使われている。

「脆弱性スキャナーはすでにある」と思っていたけど、パターンマッチング型と推論型ではそもそも得意な問題が違う。静的解析で拾えないロジック上の欠陥を見つけられるかどうか、が本質的な差だと思う。

まだ一般公開ではないので触れてない人が多いと思うけど、方向性はかなり面白い。

## 投稿3（Tips型）【約310文字】
> ソース: [Designing CLAUDE.md correctly: The 2026 architecture that finally makes Claude code work — obviousworks.ch](https://www.obviousworks.ch/en/designing-claude-md-right-the-2026-architecture-that-finally-makes-claude-code-work/)
> 推奨投稿時刻: 21:00

CLAUDE.mdが100行を超えてきたら一回見直す価値があると思う。

Claude Code開発者のBoris Cherny本人のCLAUDE.mdが約100行（2,500トークン）で、それが意図的な設計という話がある。「Claudeがすでに正しくできることは書かない」という削ぎ落とし思想。

自分も最初は「指示が多いほどちゃんと動く」と思って書き足し続けてたんだけど、ある時点からむしろ精度が落ちてる感覚があった。余計な制約が邪魔してる。

2026年の推奨構成は4層で整理するとシンプル。
CLAUDE.md（200行以内）＋.claude/skills/＋.claude/agents/＋.claudeignore

「削る」のが正解という逆転の発想、最初は受け入れにくいんだけど試してみると違いが出やすい。
