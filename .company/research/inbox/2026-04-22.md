- **ソース**: [Anthropic expands partnership with Google and Broadcom](https://www.anthropic.com/news/google-broadcom-partnership-compute) / [TechCrunch](https://techcrunch.com/2026/04/07/anthropic-compute-deal-google-broadcom-tpus/) / [Bloomberg](https://www.bloomberg.com/news/articles/2026-04-06/broadcom-confirms-deal-to-ship-google-tpu-chips-to-anthropic)

- **注目度**: 高
- **推奨投稿型**: 速報型

### 2. `claude --worktree` — CLI でネイティブ並列エージェントが動くようになった

- **日付**: 2026-04 (Boris Cherny by @bcherny on X)
- **ソース**: [Boris Cherny on X](https://x.com/bcherny/status/2025007393290272904) / [Threads投稿](https://www.threads.com/@boris_cherny/post/DVAAnexgRUj/introducing-built-in-git-worktree-support-for-claude-code-now-agents-can-run-in) / [Medium解説](https://medium.com/@naresh.kancharla/worktree-for-claude-agents-547701f82732)
- **カテゴリ**: 公式発表
- **事実**: Claude Code Desktopに既存だったgit worktree統合が、CLIにも正式移植された。`claude --worktree`（または`claude -w`）で起動すると各エージェントが独立したworktreeを持ち、同一リポジトリで並列編集しても互いに干渉しない。`--tmux`フラグを合わせると別Tmuxセッションで起動する。さらにサブエージェントのfrontmatterに`isolation: worktree`を追記するだけでエージェント単位の分離が可能。`WorktreeCreate`フックでworktree作成ロジックを完全に置き換えることもできる。
- **考察**: 「ローカルで複数Claude並列実行」はコミュニティで以前から議論されていたが、公式でネイティブ対応したことで一気にハードルが下がった。「一人で10本のPRを同時に走らせる」というユースケースをわかりやすく紹介できる。`claude -w`という短さも刺さる。
- **注目度**: 高
- **推奨投稿型**: 解説型

### 3. `/powerup` — Claude Code が自分の操作を見てインタラクティブにレクチャーしてくれる

- **日付**: 2026-04-03 (Week 14, v2.1.86–v2.1.91)
- **ソース**: [What's new Week 14 - Claude Code Docs](https://code.claude.com/docs/en/whats-new) / [Medium CLI Guide](https://medium.com/@jiten.p.oswal/the-ultimate-claude-code-cli-deep-dive-guide-to-the-newest-commands-from-april-2026-release-1eeb25b03c52)
- **カテゴリ**: 公式発表（機能追加）
- **事実**: Week 14（4/3リリース）で追加された`/powerup`コマンド。現在のプロジェクトや自分の使い方パターンを解析し、活用できていない機能をインタラクティブ形式でレクチャーする。コマンドを打つたびに「あなたはこのワークフローでこの機能を使っていないが、こう使うと効率化できる」という提案が会話形式で届く。学習コストを内側から削減するアプローチ。
- **考察**: 「Claude Codeを使いこなせていない感覚」は初中級ユーザーが抱える共通の悩み。`/powerup`はその解決策として訴求しやすい。「公式ドキュメントを読まなくてもClaude Code自身が教えてくれる」という切り口はTips型ポストの好事例。/team-onboarding（既投稿）とは「個人向け学習」vs「チーム展開」の違いで被らない。
- **注目度**: 中
- **推奨投稿型**: Tips型

### 4. 「worktreeを捨てた」— 並列エージェントの新しい分離戦略

- **日付**: 2026-04（Trigger.dev記事）
- **ソース**: [We ditched worktrees for Claude Code — Trigger.dev](https://trigger.dev/blog/parallel-agents-gitbutler) / [Shipyard multi-agent guide](https://shipyard.build/blog/claude-code-multi-agent/)
- **カテゴリ**: 技術動向 / コミュニティ
- **事実**: Trigger.devが「Claude Codeでworktreeをやめた」という記事を公開。worktreeは並列エージェントの定番手法だが、「ブランチが増え続けてGit管理が複雑になる」「worktree間でnode_modulesが重複する」「マージコンフリクトが通常より複雑になる」という課題があるとし、GitButlerベースの独自スタック（コンテナ分離 + ソフトリンク共有）へ移行した事例を紹介。一方でShipyardなどはworktreeのメリットを強調しており、コミュニティで論争に。
- **考察**: 「ベストプラクティスの正解がない」という状況は議論型ポストに最適。「worktree派 vs コンテナ派、どちらを使ってる？」という問いかけは実務者の反応を引きやすい。claude --worktree（候補2）の解説型ポストの翌日〜翌々日に「実はこういう反論もある」という議論型で出すと流れが作れる。
- **注目度**: 中
- **推奨投稿型**: 議論型
